### 第一部分：项目核心技术点总结（面试知识库）

这个项目的全貌是：**基于 RK3588 平台的工业级硬实时内核重构与 EtherCAT 底层通信优化。**

你需要掌握并向面试官展示的核心战绩分为三大块：

1.  **PREEMPT_RT 实时内核改造与冲突解决：**
    *   **原理：** 打破 Linux 原本的分时调度，开启完全抢占（Fully Preemptible），实现中断线程化。
    *   **难点（你的战绩）：** 解决瑞芯微原厂驱动与 RT 补丁的冲突。排查并修复原厂驱动中在硬中断里滥用 `spinlock` 导致的死锁/休眠问题，将其替换为 `raw_spinlock` 或移至中断下半部（Threaded IRQ）。
2.  **系统级极限调优（DTS 与 Bootargs 修改）：**
    *   **CPU 隔离与独占：** 通过 `isolcpus=7 nohz_full=7 rcu_nocbs=7`，将性能最强的 A76 核心完全隔离开来，关闭时钟滴答，作为 EtherCAT 主站的“绝对无干扰专区”。
    *   **干掉唤醒延迟：** 在设备树中注释掉 `cpu-idle-states = <&CPU_SLEEP>`，禁止 CPU 进入低功耗休眠，牺牲功耗换取微秒级的零延迟响应。
3.  **EtherCAT Native 驱动重构（极其硬核）：**
    *   **剥离 TCP/IP 栈：** 拦截 `stmmac` 的 `register_netdev` 流程，通过 `ecdev_offer` 将网卡直接注册给 EtherCAT 主站。
    *   **中断改轮询：** 屏蔽网卡的 RX/TX 硬件中断，改由主站线程以 1000Hz 频率高频主动轮询（Polling）DMA 寄存器，消除中断抖动。
    *   **零拷贝收发 (Zero-Copy)：** 绕过极其耗时的 `sk_buff` 内存分配，直接提取 DMA 净荷并通过 `ecdev_receive` 上报数据。

### 第二部分：接下来你具体需要做什么？（详细行动手册）

接下来的大约 3-4 周，请严格按照以下 6 个阶段推进，**不要跳过“踩坑”环节，那是你面试自信的来源。**

#### 阶段一：搭建纯净基线与环境（耗时：1-2天）
**目标：确保你的环境能跑通最基础的代码，排除非技术干扰。**

1.  **下载纯净内核：** 不要用野火那个打好补丁的 `lbc-develop-6.1-rt36` 分支。去拉取瑞芯微/野火**未打 RT 补丁的、普通的 Linux 6.1 分支代码**。
2.  **基线编译：** 按照官方手册，使用交叉编译工具链（aarch64-linux-gnu-gcc）编译这个纯净内核，生成 `boot.img`。
3.  **上板验证：** 将内核烧录进 RK3588 开发板，确保能正常开机，串口/SSH 能正常登录。

#### 阶段二：重现灾难，手动打 RT 补丁（耗时：3-5天）—— 【含金量最高】
**目标：亲手制造冲突，再亲手解决冲突，这就是你的实战经验。**
1.  **下载补丁：** 去 Linux Kernel 镜像站下载对应的 `patch-6.1.xx-rt36.patch.gz` 并解压。
2.  **强行打补丁：** 在纯净内核目录下执行 `patch -p1 < patch-6.1...`。终端一定会疯狂报错，生成一堆 `FAILED` 和 `.rej` 文件。
3.  **抄作业式修复：**
    *   打开野火提供的那个已经打好补丁的 SDK 源码目录。
    *   使用 **Beyond Compare** 或 **VSCode (Compare Folders)**，对比你的 `.rej` 报错文件和野火对应的 `.c` 文件。
    *   看看野火的工程师是怎么把冲突的代码改对的（重点看 `spinlock` 改 `raw_spinlock` 的地方）。
    *   手动修改你源码中的报错文件。
4.  **配置与编译：** 在 `make menuconfig` 中开启 `Fully Preemptible Kernel (RT)`。重新编译内核，直至没有报错，烧录进板子，输入 `uname -a` 看到 `PREEMPT_RT`。

#### 阶段三：EtherCAT 源码“扒皮”与逆向分析（耗时：2-3天）
**目标：把野火底层的 C 代码修改变成你脑子里的知识。**
1.  **下载官方纯净 IgH：** 弄清楚你 SDK 里 `ethercat_igh` 的版本号（如 1.5.2），去开源社区下载同版本的纯净源码。
2.  **代码比对（Diff）：** 使用 Beyond Compare 对比**纯净 IgH 目录**和**你 SDK 里的 `ethercat_igh` 目录**。
3.  **深挖驱动层：**
    *   直奔 `devices/stmmac/` 目录（纯净版里绝对没有这个）。
    *   仔细阅读 `stmmac_main.c` 和接收发送相关的 `.c` 文件。
    *   在代码里找到我前面提过的：`ecdev_offer`（设备注册）、被注释的 `request_irq`（屏蔽中断）、`ecdev_receive`（跳过 `sk_buff`）。把这些 C 代码逻辑看懂。

#### 阶段四：合并 EtherCAT 补丁与系统调优（耗时：2-3天）
**目标：将所有优化手段集成到你的系统中。**
1.  **修改设备树与启动参数：**
    *   将你提供的那份 `.patch` 里的内容（`isolcpus=7`、注释掉 `CPU_SLEEP`、修改网卡 `compatible` 为 `-ethercat` 等）手动合并到你的 `rk3588-linux.dtsi` 和对应的文本中。
2.  **编译 EtherCAT 模块：**
    *   执行你给出的那串编译命令（`./configure --enable-stmmac=yes ...` 然后 `make modules`）。
    *   生成 `ec_master.ko` 和 `ec_stmmac.ko`。
3.  **系统集成：** 将生成的 ko 文件和用户态工具（`ethercat` 命令）放入开发板，加载驱动，用命令查看主站状态，确保 MAC 地址识别成功。

#### 阶段五：性能测试与数据可视化（耗时：3-4天）—— 【简历吸睛利器】
**目标：用数据证明你的优化是有效的，秒杀其他吹水的竞争者。**
1.  **压力环境搭建：** 在开发板上运行 `stress-ng`（例如把 CPU 跑满，疯狂读写内存），模拟工业恶劣工况。
2.  **基准测试：**
    *   编译运行 `rt-tests` 里的 `cyclictest`。
    *   测试命令参考：`cyclictest -t 1 -p 99 -n -i 1000 -l 100000 -a 7`（将测试线程绑定在你隔离出来的 CPU 7 上运行，优先级设为最高 99）。
    *   记录“纯净非 RT 内核”和“你的 RT 深度优化内核”的 Max Latency（最大延迟数据）。
3.  **画图：** 把测出来的数据导出，用 Python (Matplotlib) 或 Excel 画一张**对比直方图**。普通内核的延迟可能飙到几千微秒，而你的内核稳定在 50 微秒以内。**这张图务必贴在简历项目描述的下方或面试 PPT 里。**

#### 阶段六：面试话术包装与八股文结合（耗时：最后冲刺阶段）
**目标：确保面试官问不倒你。**
围绕这个项目，复习以下 Linux 内核底层八股文，面试官一定会顺藤摸瓜问你：
1.  **中断上下文 vs 进程上下文：** 为什么硬中断里不能睡眠？`spinlock` 和 `mutex` 的本质区别？
2.  **优先级反转：** 什么是优先级反转？RT 补丁里的 `rt_mutex` 是如何通过“优先级继承”解决这个问题的？
3.  **内存管理：** 为什么在实时通信路径（如 EtherCAT 收发）里必须用 `GFP_ATOMIC` 申请内存，而不能用 `GFP_KERNEL`？
4.  **网络底层架构：** 描述一下 Linux 传统的收包流程（硬中断 -> 软中断 NAPI -> sk_buff -> 网络层）。你是怎么把这个流程“短路”掉的？
