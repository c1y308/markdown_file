# 2.FreeRTOS

## 2.1 队列

- **消息以拷贝的形式入队，而不是以引用的形式**。**（安全大于性能）**
  - 如果传递引用，data 在函数栈上，函数返回后内存失效！
  - 拷贝机制让发送者可以立即释放资源（消息变量），而不必担心接收者使用时的内存安全问题。
  - 由于是拷贝，发送后对原数据的修改不会影响队列中已存储的消息。

- 为什么传入的是**指针常量**？

  - `void*` 是一个通用指针，可以指向任何类型的数据。
  - 使用`const void*`承诺不会通过这个指针修改源数据（只读）。
  - **在函数内部使用`memcopy`来完成拷贝。**

- 为什么在FreeRTOS中xQueueSendToFront()不能在中断中使用：

``` c
// 普通版本的可能行为
BaseType_t xQueueSendToFront(QueueHandle_t xQueue, 
                            const void *pvItemToQueue,
                            TickType_t xTicksToWait) {
    
    // 如果队列满了...
    if(队列满) {
        if(xTicksToWait > 0) {
            // 普通版本可能将任务置于阻塞状态！
            vTaskPlaceOnEventList(&xQueue->xTasksWaitingToReceive, xTicksToWait);
            taskYIELD(); // 引发任务调度！
        }
    }
    
    // 在中断中任务调度是绝对不允许的！
}
```

而在中断版本中：不阻塞，直接返回队列已满错误代码 `errQUEUE_FULL` 。

# 3.硬件

## 3.1 Flash，RAM，EEPROM分别在什么地方使用？
RAM - 随机存取存储器
- 特性：**易失性存储器**。读写速度最快，但断电后所有数据都会丢失。
- 作用：作为设备的**主内存**，用于**临时**存放正在运行的程序代码和数据。
- 使用场景：
  - **程序运行空间**：当设备启动后，系统会把需要运行的代码从Flash中加载到RAM中执行，因为RAM的速度远快于Flash。
  - **变量存储**：程序运行过程中产生的全局变量、局部变量、函数调用栈等都存放在RAM中。
  - **数据缓存**：作为CPU处理数据时的缓冲区，例如处理图像、音频等大量数据时。


**简单说：凡是程序“正在”用的东西，几乎都在RAM里。**

---
Flash - 闪存
- 特性：非易失性存储器。**断电数据不丢失**。容量大，成本相对较低。但**写入速度比RAM慢**，且有擦写次数限制（通常十万到百万次）。
- 作用：用于**永久存储大量数据，如系统程序、用户文件等**。
- 分类与使用场景：
  - NOR Flash：特点：**支持XIP，即CPU可以直接从NOR Flash上取指令执行，无需先加载到RAM**。读取速度快，但写入速度慢，容量较小，价格较贵。
  使用场景：主要用于存储启动代码（Bootloader） 和可靠性要求极高的固件（如网络设备、工业控制器的核心固件）。因为系统刚上电时，RAM还是空的，CPU可以直接从NOR Flash开始执行启动程序。
  - NAND Flash：
  特点：容量大，成本低（我们熟悉的U盘、SSD基本都是NAND Flash）。但不支持XIP，必须以“块”为单位进行擦写。
  使用场景：手机/电脑的SSD/U盘：存储操作系统、应用程序、照片、视频等所有用户数据。**嵌入式设备的文件系统：存放整个操作系统内核、根文件系统、应用程序等。**

**简单说：需要“永久”保存的程序和数据，都放在Flash里。**

---

EEPROM - 电可擦可编程只读存储器

- 特性：**非易失性存储器。最大特点是可以按字节进行擦除和改写**，非常灵活。但容量很小（通常几KB到几百KB），速度慢，成本高（按字节算比Flash贵很多）。

- 作用：用于**存储那些需要频繁修改但数据量很小的配置参数**。

- 使用场景：

  - 设备配置参数：Wi-Fi路由器的SSID和密码、电视的频道设置、电子秤的校准数据等。

  - 产品序列号、用户设置：比如微波炉的预设程序、洗衣机的常用模式。

  - 运行计数：记录设备的总运行时长、开关机次数等。

**简单说：EEPROM是用来存那些“小零碎”的、需要掉电记住且偶尔会改动一下的设置信息。**
